## Task 4 projection-ellipse diagnostic for the MINGO00 vs MINGO01 discrepancy
---

### Prompt / motivation
The working prompt for this note was:

> I am studying the mingo00 and the mingo01. In Task 4 the X and Y histograms are more peaked at the center in the data than in the simulation. Before going upstream to Task 3, Task 2 and Task 1, build a new Task 4 diagnostic using the TimTrack `Xproj` and `Yproj`: fit the contour plot of the 2D histogram with ellipses, save the semiaxis ratio in the specific metadata CSV, and make this new contour+ellipse plot essential. Document the prompt and the results because this may be locating the problem.

### What was implemented
Task 4 now has a new essential plot family, `timtrack_projection_ellipse_contours`.

It does the following:

- Builds the `(xp, yp)` density from the TimTrack projections.
- Smooths the 2D histogram with a configurable Gaussian kernel.
- Fits an elliptical-Gaussian model from the density moments.
- Draws the data contours at `25%`, `50%`, and `75%` of the local peak.
- Draws the matching ellipse family at those same peak fractions.
- Writes the derived quantities into `task_4_metadata_specific.csv` with the prefix `timtrack_projection_ellipse_tt_*`.

The main metadata quantities are:

- `fwhm_axis_ratio_major_over_minor`
- `fwhm_halfwidth_xproj`
- `fwhm_halfwidth_yproj`
- `fwhm_halfwidth_x_over_y`
- `rotation_deg`

This is meant to answer a narrow question: is the projection cloud already anisotropic inside Task 4, before looking at the final `x`, `y`, and `phi` plots?

### Quick results
For a quick station-to-station comparison, I used the latest available Task 4 outputs and back-calculated

`xp = tan(theta) * cos(phi)`

`yp = tan(theta) * sin(phi)`

from the stored `theta` and `phi`. This was done only for the notebook comparison, because it lets both stations be compared uniformly without disturbing the live queue. The live Task 4 plot itself now computes directly from `tim_xp/tim_yp` (or `xp/yp`) before output pruning.

Interpretation:

- `MINGO00` is close to circular in projection space. The fitted FWHM `x/y` ratio stays near `1`.
- `MINGO01` is not close to circular. The fitted FWHM `x/y` ratio is consistently `0.78-0.85`, and the principal-axis ratio is `1.18-1.29`.
- In `MINGO01` the fitted ellipse angle is near vertical (`~84-89 deg`) for the strongest cases, so the distortion is almost aligned with the `xp` / `yp` axes rather than being a generic rotated skew.

### What this does and does not show
What this shows:

- The asymmetry is already visible in Task 4 projection space.
- The effect is not restricted to one plane combination; it appears in both 3-plane and 4-plane cases.
- The strongest visible case is the `1234` population in `MINGO01`.

What this does not show yet:

- It does not prove whether the root cause is a wrong `x` scale (`tdif_to_x` or equivalent), an upstream filtering bias in Task 1/2/3, or a geometry/efficiency mismatch.
- It does not by itself separate real-data effects from simulation mismodelling. It only says that the shape difference is already present at the Task 4 projection level.

So the useful conclusion is: this diagnostic localizes the discrepancy to at least Task 4-or-earlier, but the next step still has to be an upstream check in Task 3, Task 2, and especially Task 1.

## Task 4 time series suggests an efficiency-driven edge loss in `x`
---

### Figure
The time-series figure below was copied from the ancillary projection-axis study so this notebook keeps a stable snapshot of the evidence.

Source figure at the time of writing:

`MASTER/ANCILLARY/SIMULATION_TUNING/PROJECTION_AXIS_MISMATCH/OUTPUTS/task4_projection_ratio_timeseries.png`

![Task 4 projection ratio time series](FIGURES/2026_04_07_task4_projection_ratio_timeseries.png)

### Observation
This figure makes two things visible at once:

- the global `x` scale factor needed by the real data stays above the simulation reference band;
- the `FWHM x/y` ratios, especially for `1234`, stay below the simulation reference band.

This is not just a single bad file. The effect is systematic across the current `MINGO01` sample, although its strength changes with time.

### Working interpretation
The current reading of this figure is that the required global `x` scaling is strongly coupled to the effective plane efficiency.

A plausible mechanism is:

- the real detector starts losing efficiency first at large `|x|`, that is, near the strip / active-area edges;
- the center of the plane remains more efficient than the edges;
- then the surviving sample becomes more concentrated near the center in `x`;
- this narrows the reconstructed `x` distribution relative to `y`;
- and Task 4 responds by preferring a larger global `x` scale factor.

In that picture, the apparent `x` problem is not necessarily a pure algebraic mistake in the Task 4 reconstruction itself. It may instead be the visible consequence of an upstream efficiency / acceptance distortion that removes edge events before the fit stage.

### Why this matters
This is exactly the scenario where one must be careful before changing the `x` conversion globally.

If the origin of the effect is an efficiency map that is higher in the center and lower near the edges, then forcing a global `x` correction could partially hide the symptom while leaving the real acceptance problem untouched. It might even need to be reverted later.

So this figure should be treated as evidence that:

1. the mismatch is systematic in time, not isolated to one file;
2. the mismatch is consistent with edge-first efficiency loss in `x`;
3. any permanent `x`-scale change should be cross-checked against upstream efficiency and filtering studies before being frozen.
