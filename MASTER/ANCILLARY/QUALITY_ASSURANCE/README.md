# QUALITY_ASSURANCE

This directory contains the Stage 1 quality-assurance workspace for MINGO station metadata.

The goal is to inspect metadata generated by each Stage 1 task, visualize key columns, and then define objective `pass` / `not-pass` criteria based on observed behavior.

## Data Source

Inputs come from per-station metadata CSV files under:

`STATIONS/MINGOXX/STAGE_1/EVENT_DATA/STEP_1/TASK_N/METADATA/`

Examples:
- `.../TASK_1/METADATA/task_1_metadata_calibration.csv`
- `.../TASK_2/METADATA/task_2_metadata_filter.csv`
- `.../TASK_3/METADATA/task_3_metadata_trigger_type.csv`
- `.../TASK_4/METADATA/task_4_metadata_specific.csv`

## What We Analyze

Metadata types mainly split into:
- Execution-oriented: `_execution.csv`, `_status.csv`, `_profiling.csv`
- Data-analysis-oriented: `_calibration.csv`, `_filter.csv`, `_trigger_type.csv`, `_specific.csv`

Current QA steps in this directory focus on the analysis-oriented metadata.

## Directory Structure

- Shared config (single source of truth):
  - `config.yaml` (stations, station tree, shared static defaults)
  - `config_runtime.yaml` (date range, timestamp priority, plotting runtime defaults)
- `STEP_1_CALIBRATIONS/`
- `STEP_2_FILTERING/`
- `STEP_3_TRIGGER_TYPES/`
- `STEP_4_SPECIFIC/`

Each `STEP_*` contains:
- `TASK_1` ... `TASK_5`
- task configs (`config.yaml`, optional `config_quality.yaml`) for task-specific overrides only
- column manifests (`task_*_*_columns.txt`)
- outputs in `TASK_*/STATIONS/MINGOXX/OUTPUTS/{FILES,PLOTS}`

`run_all_steps.sh` executes all `STEP_*/run_tasks_*.sh` runners.

## Plot-First Workflow (Before Quality Criteria)

Before defining thresholds, the workflow is:
1. Read metadata CSVs.
2. Select columns from config (using the column manifest as reference).
3. Plot selected columns to inspect distributions, trends, and anomalies.
4. Use findings to later formalize `pass` / `not-pass`.

## Plot Configuration Architecture

The plotting selection is config-driven and intentionally simple:

- Use YAML lists (`[]`) to define columns that should be plotted together in the same figure.
- Allow shell-glob style patterns in the config (for example: `raw_tt_*_rate_hz`, `P[1-4]_s[1-4]_Q_*`).
- Expand patterns against the real columns present in each metadata CSV.
- Build each figure from the resolved column group.
- Compute subplot layout automatically as `n x m` to fit the group size (smallest practical grid), instead of forcing a fixed layout in the config.
- Default x-axis is filename-derived datetime from `filename_base` (pattern `miXXYYDDDHHMMSS`, where `XX=station`, `YY=year`, `DDD=day-of-year`).
- Optional override allows plotting against another column (for example number of events in file).

Example (concept):

```yaml
plot_groups:
  - name: trigger_rates
    columns: [raw_tt_*_rate_hz, clean_tt_*_rate_hz]
  - name: cleaning_ratios
    columns: [raw_to_clean_tt_*_rate_hz]

x_axis:
  mode: filename_timestamp   # default
  filename_column: filename_base
  # mode: column
  # column: P1_s1_entries_original
```

This keeps the plotting layer flexible while preserving reproducibility through versioned config files.

## Current Status of Pass/Not-Pass

Task scripts currently scaffold pass outputs (`*_pass.csv`) with configurable default values.  
Final QA criteria are intentionally deferred until exploratory plotting is complete and reviewed.
