## Task 1 filtering logic after channel-level redesign
---

### Why Task 1 is still channel-level even when it uses combinations
The filtering logic in Task 1 is now intentionally centered on **channel information**,
but it does not pretend that noise is independent channel by channel.

The key point is:

- the saved data product of Task 1 is still the channel table
- the primary objects being corrected are still the individual `Q` and `T` values of each channel
- but the decision about whether a channel value is trustworthy can legitimately use
  **cross-channel combinations**, because noise often appears correlated with specific
  channel combinations and with specific trigger-type patterns

So the analysis is still channel-level in output, while accepting that the detector noise
is not fully separable task by task.

### Task 1 now has two filters
Task 1 should be understood as having only two effective filters:

1. **Per-channel consistency filter**

For each channel, `Q` and `T` must be jointly valid:

- if both are non-zero, the channel is kept
- if one is zero and the other is non-zero, both are forced to zero
- if both are already zero, nothing changes

This regularizes the channel representation before any higher-order consistency checks.

2. **Channel-combination coherence filter**

After the per-channel regularization, Task 1 evaluates **cross-plane channel combinations**
using ancillary derived quantities:

- `Q_sum`
- `Q_dif`
- `T_sum`
- `T_dif`

These quantities are computed only for the purpose of filtering. They are **not**
the product of Task 1 and do not need to be persisted in the Task 1 parquet.

The filter is applied only when **both participating channels have non-zero `Q` and `T`**.
If a pair fails the configured limits, both participating channels are zeroed.

This means the second filter is still acting on channel values, but using pairwise coherence
to detect noise that would pass a purely independent channel treatment.

### Why this is preferred over a fully compartmented task split
A perfectly isolated task-by-task filtering philosophy would say:

- Task 1 only looks at each channel alone
- Task 2 only looks at strips
- Task 3 only looks at planes
- Task 4 only looks at tracks

In practice, that is too rigid for noisy data. Correlated noise enters through:

- channel combinations
- plane combinations
- trigger-type-dependent topologies

If Task 1 ignores those correlations completely, it can preserve channel values that are
individually within range but obviously inconsistent once combined with the rest of the event.

So the adopted logic is:

- keep each task focused on its output level
- but allow it to use the **minimum correlated information needed** to reject bad data early

For Task 1, that means:

- save channel information
- regularize channel values directly
- allow channel-pair derived quantities to decide whether those channel values are coherent

### Interpretation of later tasks under this logic
This does **not** mean the whole filtering is being duplicated downstream.
It means:

- **Task 1** uses channel data plus channel-combination coherence
- **Task 2** uses strip observables and strip-level consistency
- **Task 3** uses plane-level objects
- **Task 4** uses track-level objects

Each task remains defined by the level of data it outputs, not by a strict ban on using
correlated information from that same level.

### Practical consequence
The guiding rule is:

> We should not force a totally compartmented analysis task by task when the noise itself
> enters correlated with specific combinations and trigger types.

That is the reason the current Task 1 design is considered acceptable:

- it is channel-level in what it stores
- but it is combination-aware in how it protects channel quality
